せどりカート奪取問題

大和書房の書籍はロングテールで売れ続ける良書が多数。しかしAmazonでは、せどり業者にカートを奪われ、販売機会を逸失しています。

カート（Buy Box）を
せどりに奪われる

古書店・ブックオフで安く仕入れた書籍を「新品」として出品。定価で売る公式が価格競争で不利に。

対象: 数十銘柄

手動メールの
「もぐら叩き」状態

本部長が目視で発見 → Amazonに1件ずつメール → 排除 → また別の業者が出品。終わらない。

確認済み: 2/28ヒアリング

損失額が
見えていない

「かなりある」は感覚値。何冊のカートが取られ、いくらの売上を逸失しているか定量化が必要。

まず: 見える化が最優先
大和書房（公式）が定価で出品
↓
せどり業者が安価で相乗り出品
↓
Buy Box アルゴリズム: 価格 × 評価 × 配送
↓
せどりがカート獲得 → 公式の販売機会が失われる
重要な確認事項: 貴社は「ブランド登録はした」と認識されていますが、スポンサーブランド広告のための登録とAmazon Brand Registry（ブランド保護プログラム）は別物です。Brand Registryに正式参加しているかの確認が最優先です。

Section 03

商標登録ガイド — 何を、どう登録すべきか

Amazon Brand Registryを利用するには商標登録が前提です。大和書房にとって最適な登録戦略を整理しました。

✅

確認済みの商標

「だいわ文庫」は登録済みです（登録番号: 4976928、第16類、2006年登録、存続中）。さらに2026年1月に第9類（電子出版物）・第41類（教育サービス）で追加出願中。

⚠

「大和書房」の商標登録が必要

単行本は「だいわ文庫」レーベルではなく「大和書房」名義で出版されています。単行本のせどり対策には「大和書房」での商標登録 → Brand Registry申請が必要です。

理由: Brand Registryは「ブランド名が商品に表示されている」ことを求めます。背表紙・奥付に記載されている出版社名（大和書房）がブランド名として認められます。

商標名区分カバー範囲優先度ステータス
大和書房第16類全単行本・新刊最優先未登録 → 要出願
だいわ文庫第16類文庫シリーズ登録済み登録4976928
大和書房第9類電子書籍次点将来的に検討
商標登録の手順（特許庁への出願）

出願方法は3つ:

  • ① 弁理士に依頼（推奨）— 費用: 約5〜10万円（手数料+印紙代）。手続きを全て代行
  • ② 自分で電子出願 — 費用: 約1.2〜3万円（印紙代のみ）。特許庁 電子出願ソフトを使用
  • ③ 紙で出願 — 費用: 約1.5〜3.5万円（印紙代+電子化手数料¥2,400）

費用の内訳（第16類・1区分の場合）:

  • 出願料: ¥3,400 + ¥8,600 = ¥12,000
  • 登録料（審査通過後）: ¥32,900（10年分）
  • 審査期間: 約6〜8ヶ月（ただしBrand Registryには出願中で申請可能）
「大和出版」との混同リスクについて

「大和出版」は大和書房とは別の出版社です。Amazon上で消費者が混同する可能性があり、商標登録しておくことで区別が明確になります。

商標審査では類似商標が存在すると拒絶される場合がありますが、「大和書房」と「大和出版」は名称が異なるため、共存登録が可能と考えられます。

ニース分類（区分）の解説

商標は「どの種類の商品・サービスに使うか」を区分（クラス）で指定します。

区分内容対象
第16類印刷物紙の書籍・雑誌 → 最優先
第9類電子出版物電子書籍・ダウンロード可能な出版物
第41類教育・娯楽出版サービス・セミナー・イベント

推奨: まず第16類のみで出願（費用最小）。Kindle版も保護したい場合は第9類を追加。

商標登録の確認方法（J-PlatPat）

既存の商標登録はJ-PlatPatで無料で検索できます。

  • URL: https://www.j-platpat.inpit.go.jp/
  • 操作: トップページ → 「商標」 → 「商標検索」 → キーワードを入力
  • 24時間利用可能、ユーザー登録不要
キーワード確認目的
大和書房自社名が登録済みか・他社に取られていないか
だいわ文庫登録済み区分・存続期間の確認
大和出版競合他社の登録状況（混同リスクの把握）
出願中でもBrand Registryに申請できる仕組み

Amazon Brand Registryは2024年から出願中（pending）の商標でも申請を受け付けるようになりました。

  • 商標出願後に取得する「出願番号」をBrand Registry申請フォームに入力
  • 審査は通常2〜4週間で完了
  • 商標が正式登録されるのを待たずにブランド保護機能を利用開始できる

つまり: 商標を出願したその日にBrand Registry申請が可能です。

Section 04

Amazon Brand Registry — 根本対策

Amazonの公式プログラムでせどり業者の参入障壁を構造的に上げます。費用は無料です。

🛡

Brand Registryの最大の効果

ブランド登録すると、他の出品者が出品する際に卸売請求書（180日以内・10点以上）の提出を義務付けることができます。個人せどり業者は書類を用意できないため、事実上出品不可になります。

2024年10月以降、複数の日本の出版社が導入済みです。

Brand Registry で使える6つの機能

機能説明効果
出品承認制他セラーに卸売請求書の提出を要求最高
侵害報告ツール1クリックでせどり出品を通報。ML学習で自動ブロック進化高
IP Alert侵害疑いの出品を毎日自動通知高
A+コンテンツ商品ページの編集権限独占。検索上位にも影響中
ブランドストア大和書房専用の販売ページを構築中
Transparency個品ごとにQRコードを付与。偽造品を完全排除高
1つのアカウントで複数ブランドを管理できますか？

はい、1つのBrand Registryアカウントで複数のブランドを管理できます。

「だいわ文庫」で先行登録し、後から「大和書房」を追加登録することが可能です。それぞれのブランドに対して個別に保護機能を設定できます。

Brand Registryのデメリット・注意点
  • 古書店・個人の合法的な再販も制限してしまう場合がある
  • Seller Centralフォーラムで批判的な議論もある
  • ただし既に複数の出版社が導入済みで、業界トレンドとしては正当

対応方針: 古書市場への影響も認識した上で、新品せどり対策として誠実に活用します。

Amazonに直接確認が必要な事項
項目確認説明

要件の前に考えるべきこと — 7つの論点

在庫監視の仕組みを作る前に、ヒアリング内容に対して検討すべき重要な論点を整理しました。

1
「Amazonのページから拾う」はスクレイピング = 規約違反
結論: APIを使うべき。スクレイピングは絶対ダメ。
  • Amazon利用規約でスクレイピングは明確に禁止
  • 検出されるとVendor Centralアカウント停止リスク
  • 正規ルート: SP-API（Selling Partner API）= 無料、公式、Vendor Central対応
  • 補助ルート: Keepa API = 月€49〜で公開データ（Buy Box・出品者情報）を取得
2
「2日前の情報しか知れない」の真因は何か？
仮説: Vendor Central標準レポートの更新頻度が原因
  • Vendor Central標準レポート = 週次・月次 → 遅い
  • Real-Time Inventory Reportを使えば60〜120分遅延に改善可能
  • ただしSP-APIの開発者登録が必要（貴社側のアクション）

→ 確認事項: 現在どのレポートを使っているか？ 取次からのデータ？

3
「在庫0」と「カート落ち」は別問題 — 混同していないか？
最重要の論点。2つの状態を明確に分けるべき。
  • 在庫0 → カート落ちに見える: 在庫切れだからカートに表示されない（本当の品切れ）
  • 在庫あり → カート落ち: せどりに取られた（ブランド保護の問題）

→ 2つのデータソースが必要: SP-API（正確な在庫数）+ Keepa（Buy Box状態）

4
2,000点を手動ISBNチェックは物理的に不可能
バッチ処理で全件自動チェックすべき。手動ゼロ化が目標。
  • SP-API: 1回のリクエストで全ASIN取得可能（レポート生成→ダウンロード）
  • Keepa API: 100 ASINまでバッチクエリ可能 → 2,000点 = 20回で完了（約5分）
  • 毎日自動実行 → Google Sheetsに書き出し → アラート通知
5
「売上と在庫は別々で落としている」は統合すべき
SP-APIなら同じパイプラインで取得・結合可能。
  • GET_VENDOR_REAL_TIME_INVENTORY_REPORT — 在庫数（時間単位）
  • GET_VENDOR_REAL_TIME_SALES_REPORT — 売上数（時間単位）
  • ASINキーで結合すれば「在庫 ÷ 日販 = 残日数」を自動計算
  • 残日数 < 閾値 → アラートが実現可能
6
discover社に「リスト作成で数万円/回」は高すぎる
自動化すれば月約8,000円で毎日自動生成。
  • 現状: discover社に依頼 → 数万円/回 → 手動リスト
  • 提案: Keepa API Plan月約8,000円 + Python自動化 → 毎日自動更新
  • discover社には広告「運用」に専念してもらい、データ取得・分析は内製化
7
A+コンテンツ・バナー自動生成はスコープを分けるべき
在庫監視とは別プロジェクト。混ぜると進捗が遅くなる。
  • A+コンテンツ → Brand Registry登録が前提（商標登録と連動）
  • バナー自動生成 → 書影データ + テンプレート + 画像生成
  • 優先順位: Phase 1: 在庫監視 / Phase 2: A+ / Phase 3: バナー

ISBN → ASIN変換について

日本の書籍はISBN-10がそのままASINとして使われるケースが多いです。

  • 貴社が持つのはISBNリスト（出版物の場合）
  • ISBN-13 → ISBN-10変換が必要な場合あり
  • Keepa APIはISBN-13でも直接検索可能
SP-API開発者登録の手順

SP-APIを使うには、貴社のVendor Central内で開発者プロファイルを作成する必要があります。

  • 手順: Vendor Central → Integration → API Integration → 開発者プロファイル作成
  • リスク: 社内IT担当者の協力と承認が必要
  • 回避策: Phase 1はKeepa APIだけで始め、SP-APIはPhase 2に回す
Keepa APIだけでMVPは可能か？

可能です。ただし在庫の「正確な数量」は取れません。

  • Keepa → 在庫「あり/なし」+ Buy Box保持者 + 出品者数 + 価格
  • SP-API → 在庫「正確な数量」（sellable units）
  • MVP段階では「在庫あり/なし + カート状態」がわかれば十分
  • 「残日数」計算にはSP-APIの正確な在庫数が必要（Phase 2）
📋 SP-API 開発者登録の具体的手順（貴社側の作業）

所要時間: 約1〜2時間（本人確認の承認待ちを除く）

STEP 1: 前提条件の準備

  • 有効なVendor Centralアカウント（amazon.co.jp）
  • AWSアカウントの作成（無料枠で可。IAMユーザー作成が必要）
  • 自社のプライバシーポリシーURL（サイトにない場合は作成が必要）

STEP 2: AWSでIAMユーザーを作成

  • AWSコンソール → IAM → ユーザー → 「ユーザーの作成」
  • ユーザー名を入力し、ARN（Amazon Resource Name）を控える
  • 「セキュリティ認証情報」タブ → アクセスキーIDとシークレットアクセスキーを生成
  • 注意: これらは一度しか表示されません。即座に保存してください
  • IAMポリシー（execute-api:Invokeを許可）を作成し、ユーザーにアタッチ

STEP 3: ソリューションプロバイダーポータルで開発者登録

  • ソリューションプロバイダーポータルにVendor Centralの認証情報でサインイン
  • 本人確認を完了（約20分）
  • 開発者プロファイルを作成: 組織名、URL、連絡先、プライバシーポリシーURLを入力
  • 「プライベート開発者」として登録（自社利用の場合）

STEP 4: アプリケーション登録 & LWA認証情報の取得

  • ポータルの「アプリの開発」 → 「新しいアプリを追加」
  • アプリケーション名を入力、タイプは「プライベート」を選択
  • 必要なロール（在庫・売上レポート用）にチェック
  • 登録完了後、LWA クライアントIDとクライアントシークレットが発行される

STEP 5: 自己認可（リフレッシュトークン取得）

  • Vendor Central → 「Integration」 → 「API Integration」
  • 作成したアプリの「認証」を選択
  • リフレッシュトークンが発行される（一度しか表示されない！）
  • このトークンを開発者に安全な方法で共有

⚠ よくあるつまずきポイント:

  • クライアントシークレットは180日ごとに更新が必要（忘れるとAPI停止）
  • リージョン設定ミス: 日本は fe（Far East）リージョン
  • ロール不足: レポートAPIにアクセスできない場合は適切なロールが付与されているか確認
  • 設定変更の反映に1時間程度かかる場合がある

discover社が管理している場合:

  • discover社がVendor Centralの管理者権限を持っている場合、上記STEP 2-5の一部をdiscover社に委任可能
  • ただしAPIキーの管理権限は大和書房側で保持することを推奨
  • 代理店にアクセスさせる場合は「パブリックアプリケーション」のOAuth認証フローを使用
🔑 Keepa API 利用開始の手順（貴社側の作業）

所要時間: 約15分 / 費用: €49/月〜（約8,000円）

重要: Keepaには2種類の有料プランがあります
• Data Access（€19/月） = ブラウザ拡張機能用。APIは1トークン/分しか発生せず、自動クエリには使えません
• API Plan（€49/月〜） = システム連携用。2,000件/日の自動クエリにはこちらが必須

STEP 1: Keepaアカウント作成（貴社）

  • keepa.com にアクセス
  • 右上の「Register」 → メールアドレス・パスワードを入力して登録
  • メールアドレス認証を完了

STEP 2: API Planに申し込み（貴社）

  • keepa.com/#!api にログイン状態でアクセス
  • 「PRICING PLANS」セクション → API Plan 20（€49/月）を選択
  • 「SUBSCRIBE」 → 請求先情報を英語で入力
  • クレジットカード or Google Payで決済
  • 年払いオプションはなし（月払い固定、通貨はユーロ固定）

STEP 3: APIキーを取得して開発者に共有（貴社 → 開発者）

  • keepa.com/#!api の「YOUR ACCESS」セクション
  • 「Private API access key」をコピー
  • 開発者に安全な方法で共有（パスワードマネージャー等推奨）
  • キーは再発行可能（アカウント設定から）
