分析SQLクエリ（4本）

クエリ1: 書店別返品率ランキング 
-- クエリ1: 書店別返品率ランキング
SELECT
  m.書店名,
  m.クラスター,
  SUM(r.返品部数) / SUM(t.配本部数) AS 返品率,
  SUM(t.配本部数) AS 合計配本部数,
  SUM(r.返品部数) AS 合計返品部数
FROM 配本トランザクション t
JOIN 返品トランザクション r USING(書店ID, 書籍ID)
JOIN 書店マスタ m USING(書店ID)
WHERE t.配本日 >= DATE_SUB(CURRENT_DATE(), INTERVAL 6 MONTH)
GROUP BY m.書店名, m.クラスター
ORDER BY 返品率 DESC;
クエリ2: 重版チャンス書籍抽出 
-- クエリ2: 重版チャンス書籍抽出（残週数4週未満）
SELECT
  b.タイトル,
  b.著者,
  b.現在在庫数,
  SUM(p.販売部数) AS 先週販売数,
  b.現在在庫数 / NULLIF(SUM(p.販売部数), 0) AS 残週数
FROM 書籍マスタ b
JOIN POSデータ p USING(書籍ID)
WHERE p.週 = (
  SELECT MAX(週) FROM POSデータ
)
GROUP BY b.書籍ID, b.タイトル, b.著者, b.現在在庫数
HAVING 残週数 < 4
ORDER BY 残週数;
クエリ3: オケージョン別成功率 
-- クエリ3: オケージョン別成功率
SELECT
  オケージョン,
  COUNT(*) AS 提案数,
  SUM(CASE WHEN 結果 = '成約' THEN 1 ELSE 0 END) AS 成約数,
  ROUND(
    SUM(CASE WHEN 結果 = '成約' THEN 1 ELSE 0 END) * 100.0
    / COUNT(*), 1
  ) AS 成功率_pct
FROM 営業活動ログ
WHERE 訪問日 >= DATE_SUB(CURRENT_DATE(), INTERVAL 3 MONTH)
GROUP BY オケージョン
ORDER BY 成功率_pct DESC;
クエリ4: 書店クラスター別ROI 
-- クエリ4: 書店クラスター別正味売上
SELECT
  m.クラスター,
  COUNT(DISTINCT m.書店ID) AS 書店数,
  SUM(t.配本部数) AS 合計配本,
  SUM(r.返品部数) AS 合計返品,
  SUM((t.配本部数 - r.返品部数) * b.定価) AS 正味売上円
FROM 配本トランザクション t
JOIN 返品トランザクション r USING(書店ID, 書籍ID)
JOIN 書店マスタ m USING(書店ID)
JOIN 書籍マスタ b USING(書籍ID)
GROUP BY m.クラスター
ORDER BY 正味売上円 DESC;

SECTION 12

注意点・限界

反証チェック5項目

リスク 01
取次との関係変化
書店直交渉強化が取次との軋轢を生む可能性
→ 段階的実施と取次への情報共有で回避。取次を敵にしない。
リスク 02
データ収集の実現可能性
書店がPOSデータを共有しない場合がある
→ 取次データで代替可能。書店個別データは二次的に収集する。
リスク 03
kintone入力定着
営業担当者の入力負担増で形骸化するリスク
→ テンプレート化と管理者モニタリングで対処。入力5分以内を目指す。
リスク 04
山田課長依存リスク
暗黙知移転が完了前に退職・異動する可能性
→ ヒアリングをPhase 1最優先タスクにする。記録は複数名が立ち会う。
リスク 05
書店のDX格差
小規模書店はデジタル対応不可の場合がある
→ クラスター4・5向けはアナログ維持。全書店を同じ扱いにしない。

未解決の問い（次のステップ）

  • 取次との新しい協業モデルは何か？（書店直交渉の境界線をどこに引くか）
  • 著者・編集チームとの連携はどう設計するか？（営業と編集の壁を超えた情報共有）
  • 電子書籍との連動戦略は？（紙+電子の統合データ管理と相互誘導）
  • 新刊点数増加時の営業リソースはどう確保するか？（スケールのボトルネック）
