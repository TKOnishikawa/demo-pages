Section 02

成果物詳細

各成果物のポイント・現在地・次のステップ。

① 注文書OCR自動入力（手入力削減）

背景：FAX注文書が月1,400〜2,400枚届き、松本課長を中心に手入力対応。工数負荷・誤入力・属人化が課題。

実装：大和書房社内のNucBox4（4F総務部設置）にRDP接続し、Gemini Vision API + マスタ照合パイプライン（v6.1）を構築。BEAT VPN不要で、社内PCから THINK's・共有フォルダへ直接アクセス可能な環境を実現。

現状：マスタ照合精度96%（52件中50件）。エラー傾向は0/1混同・手書き5/6混同のみ。

セッション進行：OA-1〜OA-6の6段階。現在 OA-1（全量実行）・OA-2（共有フォルダ連携）並行進行中。

次のステップ：OA-3（THINK's一括入力画面への貼付検証）、OA-5（Windows Task Scheduler 自動実行）、OA-6（本稼働 Go/No-Go）。想定 5月中に完全自動化。

② 注文書ジェネレーター（発注側 Web UI）

背景：書店側への発注書を、毎回エクセルで作成して担当者が運用。新刊・在庫でフォーマットが違う、ISBN手入力が手間、という課題。

実装：Next.js 16 + React 19 でカスタムドメイン（order.daiwashobo.omoshiku.jp）に本番デプロイ。ISBN検索→自動入力、A4プレビュー、PDF/Word書き出し、JSON保存／読込（画像Base64込み）、新刊・在庫の自動切替、タイトル縦位置調整スライダーなど追加機能を継続デプロイ中。

現状：本番稼働中。営業部が実務で使用。

次のステップ：書店リスト連携、発注履歴の Kintone 連携、社内注文入力フローとの接続。

③ Amazon BuyBox 対策（せどり監視 × 在庫見える化）

背景：稼働在庫5,882点のうち、せどり業者にBuy Boxを奪われる「カート落ち」問題。discover社への外部委託で数万円/回・2日遅延。

実装：Keepa API で日次差分更新（トークン消費12,500/日）、Email通知（カート奪取・新規RED・日次サマリー3種）、Sheets自動更新。リポ：TKOnishikawa/daiwashobo-amazon-monitor（Private）。

現状：Sprint 0 完了（基盤・スキャン・トークン管理・Email送信）。Sprint 1 で日次アラート実装中。

次のステップ：Sprint 2 で SP-API Vendor 統合（Data Kiosk API、遅延34時間）。大和書房がVendor Central管理者になるとAPI開発者登録可能。Sprint 3 で運用安定化・週次レポート。

④ Kintone × THINK's 連携（マスタ日次同期）

背景：THINK's の書籍・書店・取引先・在庫マスタを、各部署がエクセルCSVで個別ダウンロード→バラバラに保管していた。最新版が分からない問題が常態化。

実装：THINK's → Kintone の日次自動同期（PostMessage方式、バージョンアップ対応）。Kintone側を参照先として各申請アプリ（契約書・発注書・支払依頼・慶弔等）から共通参照できる基盤。

現状：技術検証完了。5月から本番投入可能。

次のステップ：4/23 の総務部MTGで「やりたいこと16項目」の優先度合意 → Phase 0 として本基盤を稼働しつつ、Phase 1 で業務アプリ群を順次構築（詳細はSection 03）。

Section 03

THINK's 連携を土台に、社内の申請・契約・支払・管理業務を Kintone に一元化していく構想です。4/23 総務部MTGで「やりたいこと16項目」の優先度合意、5月から本番構築開始。

Phase 0〜4 ロードマップ

PHASE 0  /  〜 2026-05
マスタ同期 ＋ クイックウィン（慶弔申請）
THINK's→Kintone マスタ同期の本番投入。並行して慶弔見舞金申請（出産・結婚・死亡弔慰統合）を先行実装し、横展開の型を作る。
PHASE 1  /  2026-05 〜 07
契約・発注・支払 4アプリ連携
業務委託契約 → 出版契約書類 → 発注書 → 支払依頼 の順で構築。取引先マスタの正規化で、契約申請の二度手間と支払依頼のRPA阻害要因を構造的に解消。
PHASE 2  /  2026-07 〜 09
freee ／ THINK's ／ DocuSign 自動連携
Phase 1 の4アプリを外部システム接続。BizRobo廃止。月末の手作業統合を解消し、リアルタイム連携へ。
PHASE 3  /  2026-09 〜
海外版権 ＋ 社内申請系 横展開
248列の海外版権管理エクセルを整形・Kintone化。慶弔の型から出張・有給・交通費・経費・忌引など社内申請系を量産。
4/23 総務部MTG のゴール： 「やりたいこと16項目」を優先度マトリクス（難易度×価値）に載せて、Phase 0〜3 のスコープを合意。 詳細は別資料 「Kintone 全社DX基盤 構想」 を総務部向けに作成済。
Section 04

RPA — BizRobo 代替

4/10 総務ヒアリングで、BizRobo（レイテック社）運用に関する課題が挙がりました。ロボットが動作不安定でテスト段階で停止したまま、自己管理・修正対応がしづらい。この費用を浮かせて、より便利な投資に振り替えたいご意向です。

⚠

現状の課題

・ ロボットがテスト段階で停止したまま
・ 自己管理・修正が難しい（ブラックボックス）
・ 相手先名のバラつきでRPA化不能な工程あり
・ 年間費用が固定でかかり続ける

✨

Kintone基盤での解決

・ バリデーションを先に効かせエラー発生率を下げる
・ マスタ参照で相手先名を正規化 → RPA阻害要因を解消
・ API/Webhookで直接連携（ロボット不要）
・ 浮いた費用をKintoneライセンス拡張に転用

移行戦略： 松本課長が既に着手済の Kintone注文入力アプリを土台に、支払依頼（A3）・発注書（A2）・契約書申請（A4/A6）を順次構築することで、BizRoboの担当業務を置き換えていきます。新しいロボットを作らず、「エラーが出にくい仕組み」に構造転換する方向です。
Section 05

セールスプロモーション — 営業DX

2/28・3/3 ヒアリングで井坂様・営業チームからいただいた素材を踏まえ、Kintoneポータルを中心とする営業情報の一元化から着手するイメージです。

SHORT  /  短期
Kintone ポータルで情報一元化
新刊締切・重版予定・部門横断情報を一箇所で参照できる営業ダッシュボード。Phase 1〜2 の副産物として整備可能。
MID  /  中期
初速可視化ダッシュボード（POS × Kintone）
発売直後の売れ行きを POSデータと連動して可視化。配本判断・重版判断の精度向上、機会損失の定量化。
LONG  /  長期
再販判断支援・営業ナレッジベース
営業勝ちパターン分析、SNS・LINE活用などヒアリング済素材をKintoneにナレッジ化。部門横断でノウハウを資産化。
既存基盤との接続： Amazon BuyBox 対策と注文書ジェネレーターは この営業DXの一部として機能する位置付けです。監視アラート・発注業務を Kintone 営業ダッシュボードから横断できるようにすることで、部門横断の意思決定が加速します。

Section 07

ご相談したいポイント

本日のMTGで、以下の4点をすり合わせさせてください。

  • 1
    今後の優先テーマの温度感
    Kintone 基盤／RPA 置換／セールスプロモーション の3領域。どこに比重を置くか、本日の議論で方向感を合意したいです。弊社としてはKintone 基盤 → RPA自然置換 → 営業DX統合の順がROI最大と考えています。
  • 2
    現状の支援体制（月次関わり方）の調整
    Kintone基盤構築は数ヶ月〜1年単位の継続稼働になります。月次の稼働時間・訪問頻度・優先順位を、現契約の延長 or 拡張契約でどう整理するかご相談したいです。
  • 3
    BizRobo など既存ツールの整理方針
    浮いた費用を Kintone ライセンス拡張（編集部・営業部）に振り替える方向で構想中。BizRobo の契約状態と廃止スケジュールを共有いただけると、具体的な振替額の設計ができます。
  • 4
    役割分担（オモシク／社内／他社ベンダー）
    Kintone 構築は弊社、日々の運用は社内、Amazon広告は discover社、が基本線。松本課長が着手中の Kintone 注文入力アプリとの連携方針、営業DXを誰が担うか、等を整理したいです。
