定着・活用設計 — 「使われるダッシュボード」のために

CORE PRINCIPLE

ダッシュボードの成否は「技術的完成度」ではなく「現場の意思決定を変えたか」で決まる。営業所長が毎朝5分で自分の数字を確認し、部下に的確な指示を出せるようになること。それが唯一の成功基準。

⚙ ビジネス目的の明確化（着手前に必ず合意）

  • 「何を見て、どんな判断をしたいか」を営業所長3名以上にインタビュー
  • Excel月次速報の「本当に見ている箇所」と「作っているけど見ていない箇所」を分離
  • KPI定義書（計算式・粒度・更新頻度）を先方IT担当と合意
  • ダッシュボードの想定利用シーンを3つ以上定義（月次会議、日常チェック、経営報告）

🔄 ユーザーFB × チューニングサイクル

  • パイロット期間（4週間）で週1回のFB収集（5分アンケート: 見やすさ/必要な情報/不要な情報）
  • 「一度も開いていない人」を2週目に特定し、個別に理由をヒアリング
  • v1.0 → v1.1 → v1.2 の3イテレーションをパイロット期間中に実施
  • 全営業所展開後も月1回のFB収集を3ヶ月継続
  • 利用率KPIを設定: 月1回以上アクセスした営業所長の割合 ≥ 90%

🧠 なぜNotebookLMなのか — 根本的な設計思想の違い

Gemini in Google Drive

= 汎用AIアシスタント

  • 全インターネット知識 + Driveファイルを「参考」
  • 答えが見つからない時: 知っている知識で補完する（ハルシネーション）
  • 建設現場で「この接着剤は〇〇に使えます」→ 実は使えない（安全リスク）
NotebookLM

= 専用ナレッジベース

  • アップロードした資料だけを情報源にする
  • 答えが見つからない時: 「この資料には情報がありません」と正直に回答
  • 建設現場で使っても、ソース不明の情報が混入しない

⚠️ 建設業での安全リスク

リフォーム現場では材料の耐荷重・施工手順・安全基準など「間違った情報＝事故」に直結する。Gemini in Driveは便利だが、現場ナレッジとして使うには「知らないことを知らないと言える」NotebookLMの方が安全。

定着・活用設計 — 「使われるナレッジベース」のために

CORE PRINCIPLE

ナレッジベースの失敗パターンは「ドキュメントを入れて終わり」。成功パターンは「具体的な利用シーンから逆算して、必要最小限のドキュメントを入れ、ユーザーFBでチューニングし続ける」。

⚙ ビジネス目的の定義（3つの利用シーン）

  • シーン1: 新人研修「水回りリフォームの養生手順を教えて」→ マニュアルから即回答
  • シーン2: 施工トラブル対応「壁紙の浮きが発生した場合の対処法は？」→ FAQ + 過去事例から回答
  • シーン3: 見積作成支援「キッチンリフォームの標準単価と注意点は？」→ 単価表 + ガイドラインから回答
  • 各シーンで「NotebookLM導入前の解決方法」と「導入後の期待体験」を明文化
  • パイロット参加者に「よく聞く質問TOP10」を事前にリストアップしてもらい、回答品質をテスト

📚 資料の網羅的収集 + 品質担保

  • P1（必須）: 施工マニュアル、FAQ集、安全管理マニュアル（推定30-90ファイル）
  • P2（推奨）: 見積ガイドライン、クレーム対応事例、建材カタログ（推定40-100ファイル）
  • 全ドキュメントに見出し構造（H1→H2→H3）を統一適用
  • 専門用語の定義セクションを各マニュアル冒頭に配置
  • FAQ形式化: 可能な限り「Q: 〜 A: 〜」構造に変換（NotebookLMの精度が大幅に向上）
  • 作成日・更新日を全ドキュメントに明記し、古い情報を識別可能に

定着・活用設計 — 「管理ではなく支援」のGBP運用

CORE PRINCIPLE

本部一元管理を「監視」と受け取られると支社の協力が得られない。「各営業所の集客を本部がサポートする仕組み」というフレーミングが成否を分ける。

⚙ ビジネス目的の定義

  • 目標: 37営業所のGBP経由の問い合わせを6ヶ月で5%増
  • KPI: 月間検索表示回数、クリック数、電話発信数、口コミ評価平均
  • 各営業所長に「あなたの営業所の課題と改善提案」をレポートとして月次配信
  • 口コミ返信テンプレート（ポジティブ/ネガティブ）を提供し、返信率を50%以上に

「作ったけど使われない」を防ぐ共通原則

1. 利用シーンから逆算

「こんな技術がある」ではなく「こんな場面で困っている」からスタート。先方の課題認識pptxが最良の起点。

2. パイロットで小さく始める

全社展開前に1-5拠点で検証。定着KPIを設定し、クリアしてから横展開。いきなり37営業所は危険。

3. FBループを2週間で回す

v1.0を出したら2週間以内にFB収集 → v1.1へ。完璧を目指す前に「まず使ってもらう」。使わない人に個別ヒアリング。

4. 旧プロセスとの並行運用期間

いきなり「明日からこっち」は失敗する。2ヶ月の並行運用で信頼を醸成してから旧プロセスを廃止。

5. 成功を数字で語る

「便利になった」では経営層は動かない。「年間387万円の工数削減」「問い合わせ5%増」等の定量効果を初月から計測。

6. 推進者を「中」に置く

外部コンサルだけでは定着しない。IT担当2名 + パイロット営業所長を「社内推進者」に育てる設計。

増田さん以外に巻き込むべきステークホルダー

IT担当2名

QW-1/2の構築・保守の実行者。スキルレベルと意欲の確認が最優先。「やらされ感」ではなく「自分のスキルアップにもなる」動機づけ。

支社長（2-3名）

パイロット候補の営業所長。成功体験を横展開する「アンバサダー」。意欲的な人を選ぶ。

経理部門長

QW-1のデータ整合性の担保者。基幹CSV→ダッシュボードの数字と経理の数字が一致することの確認。

現場営業（1-2名）

QW-4のワークショップに必須。「本当の業務フロー」は現場にしか分からない。

So What

4つのQuick Winはすべて投資ゼロ（または月数千円）で即効果。しかし「作れること」と「使われること」は全く別問題。技術実装と同じかそれ以上のエネルギーを「定着設計」に投資することが、DXの成否を分ける。「経営層のDX意識が弱い」というニッカホームの現状では、Quick Winの成功体験こそが中期投資への唯一のパスポート。だからこそ、1つ1つのQWを「確実に使われ、確実に効果を生む」ものにすることが最優先。
